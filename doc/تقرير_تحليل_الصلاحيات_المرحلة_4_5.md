# تقرير تحليل نظام الصلاحيات — المرحلة 4 والتهيئة الجزئية للمرحلة 5

**التاريخ:** 2025  
**النطاق:** RasRas ERP — Default Deny، جدول `path_permission`، ربط الواجهة بقواعد المسارات.

---

## 1. ملخص تنفيذي

| البند | الحالة |
|--------|--------|
| **Default Deny (PathPermissionFilter)** | ✅ مفعّل — أي مسار غير Whitelist وبدون قاعدة في `path_permission` → 403 |
| **تغطية مسارات API في `path_permission`** | ❌ **جدول فارغ** — لا يوجد Seed لقواعد API؛ جميع المسارات غير المغلفة ستُرجع 403 |
| **مسارات Whitelist** | ✅ مُعرّفة في `PathPermissionConstants`؛ **فجوة:** GET `/api/company` غير مستثنى |
| **صلاحيات الواجهة (قواعد من API)** | ⚠️ الواجهة **مهيأة** لجلب path-rules مع fallback ثابت؛ بدون قواعد في الجدول يبقى الاعتماد على `PATH_PERMISSION_PATTERNS` |
| **@PreAuthorize** | ✅ لا تزال فعّالة على جميع الـ Controllers كطبقة أمان ثانية |

**الخلاصة الحرجة:** تفعيل Default Deny مع جدول `path_permission` **فارغاً** يؤدي إلى رفض كل طلبات API (ما عدا Whitelist). يجب **عدم الاعتماد على Default Deny في الإنتاج** قبل ملء الجدول بقواعد تغطي كل المسارات المهمة، أو إعادة تشغيل التطبيق مؤقتاً بنمط Safe Mode حتى اكتمال الملء.

## 2. تحليل النظام الحالي

### 2.1 تطبيق Default Deny في PathPermissionFilter

- **الموقع:** `web/backend/.../shared/security/PathPermissionFilter.java`
- **السلوك:**
  - إذا كان المسار ضمن **Whitelist** → يسمح بالمرور دون التحقق من `path_permission`.
  - إذا كان المسار **خارج Whitelist**:
    - يستدعي `PathPermissionService.getAllowedPermissionCodesForApi(path, method)`.
    - إذا **لا توجد قواعد** للمسار → **403 Forbidden** (Default Deny).
    - إذا وُجدت قواعد → يتحقق من أن المستخدم (من `Authentication.getAuthorities()`) يملك **أي** صلاحية من المسموح بها؛ وإلا → 403.
  - الفلتر يعمل **بعد** `JwtAuthenticationFilter`، لذا الهوية معروفة عند التقييم.
- **النتيجة:** Default Deny **مطبّق فعلياً** في الكود.

### 2.2 تغطية مسارات API بجدول `path_permission`

- **التحقق من DataSeeder:** لا يوجد أي استدعاء لـ `path_permission` أو `PathPermission` في `DataSeeder.java`.
- **النتيجة:** جدول `path_permission` **لا يُملأ تلقائياً** عند التشغيل الأول؛ وهو **فارغ** ما لم تُضاف قواعد يدوياً أو عبر سكربت SQL.
- **التأثير مع Default Deny:** كل مسار API (ما عدا Whitelist) **غير مغطى** → كل الطلبات المصادق عليها ستُرجع **403** حتى تُضاف قواعد.

### 2.3 مسارات Whitelist

- **المصدر:** `PathPermissionConstants.WHITELIST_PREFIXES`
- **القيم الحالية:**
  - `/api/auth/`
  - `/api/settings/public`
  - `/api/swagger-ui`, `/api/v3/api-docs`, `/api/api-docs`
  - `/api/actuator`
  - `/api/uploads/`
  - `/api/public/`

- **مقارنة مع SecurityConfig:** في `SecurityConfig` يوجد `requestMatchers( GET, "/company").permitAll()` — أي **GET /api/company** مسموح به على مستوى Spring Security، لكن **PathPermissionFilter** لا يستثني `/api/company`. بما أن الفلتر يعمل بعد المصادقة، طلب GET /api/company من عميل مصادق سيصل للفلتر؛ وإذا لم توجد قاعدة في `path_permission` لـ `/api/company` فسيُرفض بـ 403.
- **الفجوة:** إما إضافة بادئة للـ Whitelist (مثلاً استثناء GET فقط يحتاج منطق إضافي) أو إضافة **قاعدة في `path_permission`** لـ GET `/api/company` مع صلاحية مناسبة (أو صلاحية عامة للمصادقين) حتى لا يُرفض الطلب.

### 2.4 صلاحيات المستخدمين في الواجهة وقواعد path_permission

- **الواجهة (المرحلة 5 الجزئية):**
  - تستدعي **GET /api/permissions/path-rules** عند تحميل اللوحة (`DashboardLayout` useEffect) وعند **refreshUserPermissions**.
  - `canAccessPath()` و `getRequiredPermissionForPath()` في `permissionUtils.ts`: إذا وُجدت قواعد من API (`pathRulesFromApi`) وتطابق المسار تُستخدم؛ وإلا **fallback** إلى المصفوفة الثابتة `PATH_PERMISSION_PATTERNS`.
- **الوضع الحالي:** جدول `path_permission` فارغ → **لا توجد قواعد PathType=FRONTEND** → استدعاء path-rules يُرجع قائمة فارغة → `setPathRulesFromApi(null)` أو قائمة فارغة → الواجهة تعتمد بالكامل على **PATH_PERMISSION_PATTERNS**.
- **النتيجة:** عرض القوائم والمسارات يعمل كما قبل المرحلة 5؛ لا يوجد كسر. عند ملء قواعد FRONTEND لاحقاً، الواجهة جاهزة لاستخدامها مع الاحتفاظ بالـ fallback.

**ملاحظة:** حتى يعمل **جلب path-rules** مع Default Deny، يجب وجود **قاعدة في `path_permission`** لـ GET `/api/permissions/path-rules` (أو نمط مثل `/api/permissions/**`) وربطها بصلاحية يملكها المستخدمون الذين يحتاجون تحميل القواعد (مثلاً SECTION_SYSTEM أو صلاحية عامة للمصادقين).

### 2.5 @PreAuthorize

- تم التحقق من الـ Controllers: **جميعها** تحتوي على `@PreAuthorize` (على مستوى الـ class أو الـ method) باستخدام ثوابت من `SecurityConstants` أو `isAuthenticated()` أو `hasAuthority(...)`.
- **النتيجة:** طبقة الحماية الثانية **فعّالة**؛ لا يوجد إزالة لـ @PreAuthorize.

## 3. كشف الفجوات

### 3.1 مسارات غير مغطاة في `path_permission` (ستُرجع 403 مع Default Deny)

في الوضع الحالي (جدول فارغ)، **كل** مسار API غير موجود في Whitelist غير مغطى. أبرز المجموعات:

| الفئة | أمثلة مسارات | التأثير |
|--------|----------------|----------|
| **تسجيل الدخول والعام** | /api/auth/login, /api/auth/refresh, /api/settings/public, /api/public/* | مستثناة عبر Whitelist ✅ |
| **الشركة** | GET /api/company | ❌ غير مستثنى؛ 403 للمصادقين إن لم تُضف قاعدة أو استثناء |
| **الصلاحيات والأدوار** | /api/permissions, /api/permissions/path-rules, /api/permissions/path-audit, /api/roles | ❌ غير مغطاة → 403 |
| **المستخدمون والموظفون** | /api/users, /api/employees, /api/hr | ❌ غير مغطاة |
| **المشتريات والموردون** | /api/procurement/*, /api/suppliers, /api/suppliers/invoices | ❌ غير مغطاة |
| **المخزون والعمليات** | /api/inventory/*, /api/units, /api/stock-movements | ❌ غير مغطاة |
| **المبيعات والـ CRM** | /api/sales/*, /api/crm/customers | ❌ غير مغطاة |
| **المالية والمحاسبة** | /api/finance/payment-vouchers, /api/journal-entries | ❌ غير مغطاة |
| **الاعتمادات والإعدادات** | /api/approvals, /api/approval-limits, /api/settings, /api/settings/database | ❌ غير مغطاة |
| **لوحة التحكم والرفع** | /api/dashboard, /api/upload | ❌ غير مغطاة |

**أداة التدقيق:** استخدام **GET /api/permissions/path-audit** (بعد إتاحة الوصول لها بقاعدة أو مؤقتاً) يعطي قائمة دقيقة بـ `uncoveredPaths` لكل (path, method).

### 3.2 عناصر واجهة / Sidebar لا تطابق صلاحيات المستخدمين

- **المصدر الحالي للواجهة:** `PATH_PERMISSION_PATTERNS` في `permissionUtils.ts` + `requiredPermission` في عناصر القائمة في `DashboardLayout.tsx`.
- **الوضع:** لا توجد قواعد FRONTEND في `path_permission` → الواجهة تعتمد فقط على المصفوفة الثابتة. لا يوجد **تناقض** بين جدول path_permission والواجهة لأن الجدول فارغ.
- **مخاطر لاحقة:** عند إضافة قواعد FRONTEND في الجدول دون مزامنة مع `PATH_PERMISSION_PATTERNS` أو مع عناصر السايد بار، قد يظهر رابط لمسار لا يملك المستخدم صلاحيته (أو العكس). **التوصية:** عند ملء FRONTEND، الحفاظ على تطابق بين أنماط المسارات في الجدول وأنماط السايد بار والـ Routes.

### 3.3 مسارات أو صلاحيات قد تسمح بتجاوز الوصول

- **الطبقة الحالية:** PathPermissionFilter يمنع الوصول إذا لم توجد قاعدة أو إذا المستخدم لا يملك أي صلاحية مسموح بها؛ @PreAuthorize تمنع الوصول حسب الصلاحية على مستوى الـ method.
- **مخاطر محتملة بعد الملء:**
  - **قواعد واسعة جداً:** مثلاً `/api/**` مع صلاحية واحدة قد تمنح وصولاً أوسع من المقصود. يُفضّل أنماط أوضح (مثل `/api/inventory/**`, `/api/sales/**`) وربطها بصلاحيات الأقسام المناسبة.
  - **تناقض مع @PreAuthorize:** إذا ربطت قاعدة path_permission بصلاحية أوسع من صلاحية الـ Controller (مثلاً SECTION_MAIN لـ /api/settings)، المستخدم سيجتاز الفلتر لكن @PreAuthorize سترفض — سلوك آمن لكن قد يسبب التباساً. الأفضل أن تكون صلاحية path_permission مطابقة أو أضيق من صلاحية الـ Controller.

## 4. التوصيات لإصلاح الفجوات

### 4.1 قبل الاعتماد الكامل على Default Deny

1. **ملء جدول `path_permission` لمسارات API:**
   - إما إضافة **Seed في DataSeeder** يدرج قواعد PathType=API لأنماط مثل:
     - `/api/company` (GET) → صلاحية مناسبة (مثلاً isAuthenticated أو SECTION_SYSTEM حسب السياسة)
     - `/api/permissions/**` → SECTION_SYSTEM (لـ path-audit و path-rules وقائمة الصلاحيات)
     - `/api/roles/**`, `/api/approval-limits/**` → SECTION_SYSTEM
     - `/api/settings/**` (عدا public) → SECTION_SYSTEM
     - `/api/users/**` → SECTION_USERS أو SECTION_SYSTEM حسب الـ endpoint
     - `/api/employees/**`, `/api/hr/**` → SECTION_EMPLOYEES
     - `/api/procurement/**`, `/api/suppliers/**` → SECTION_PROCUREMENT (أو الصلاحيات الدقيقة حسب التوثيق)
     - `/api/inventory/**` → SECTION_WAREHOUSE / SECTION_OPERATIONS / INVENTORY_VIEW حسب المسار
     - `/api/sales/**` → SECTION_SALES
     - `/api/crm/**` → SECTION_CRM
     - `/api/finance/**` → SECTION_FINANCE
     - `/api/journal-entries/**` → ACCOUNTING_* حسب الفعل
     - `/api/dashboard/**` → isAuthenticated
     - `/api/approvals/**` → SECTION_MAIN / AUTHENTICATED حسب التوثيق
     - `/api/upload/**` → حسب السياسة (مثلاً authenticated)
   - أو تشغيل **path-audit** بعد إضافة أول مجموعة قواعد ثم إكمال الناقص حسب `uncoveredPaths`.

2. **معالجة GET /api/company:**
   - إما إضافة استثناء في الفلتر لـ GET `/api/company` (إن كان يجب أن يبقى عاماً للمصادقين)،
   - أو إضافة قاعدة في `path_permission` لـ GET `/api/company` مع صلاحية يملكها من يُفترض أن يصل (مثلاً isAuthenticated لا تُعبّر عنها بصلاحية في الجدول؛ يمكن استخدام صلاحية مشتركة مثل SECTION_MAIN أو إنشاء صلاحية AUTHENTICATED في الجدول وربطها بكل الأدوار).

3. **التحقق بعد الملء:** استدعاء **GET /api/permissions/path-audit** والتأكد من أن `uncoveredPaths` فارغة (أو تحتوي فقط مسارات مقصودة كتجريبية).

### 4.2 المرحلة 5 الكاملة (ربط الواجهة بالقواعد)

1. **دمج جلب قواعد API في الواجهة:** ✅ موجود — `fetchPathRules()` و `setPathRulesFromApi()` واستدعاؤهما عند تحميل اللوحة وعند `refreshUserPermissions`.
2. **تحديث canAccessPath لتعمل ديناميكياً:** ✅ موجود — استخدام `pathRulesFromApi` عند التوفر مع fallback إلى `PATH_PERMISSION_PATTERNS`.
3. **ملء قواعد path_permission:**
   - **API:** كما في 4.1 (بما فيها GET /api/permissions/path-rules و GET /api/company).
   - **FRONTEND:** إضافة قواعد PathType=FRONTEND لأنماط مسارات الواجهة (مثل `/dashboard/**`, `/dashboard/settings/**`, …) وربط كل نمط بصلاحية واحدة أو أكثر مطابقة لـ `PATH_PERMISSION_PATTERNS` والتوثيق؛ بحيث GET /api/permissions/path-rules يُرجع قائمة تستخدمها الواجهة دون كسر العرض.
4. **اختبارات إضافية:**
   - اختبار تكامل: بعد ملء الجدول، استدعاء path-audit والتحقق من coveredCount = totalMappings (بعد استبعاد Whitelist).
   - اختبار يدوي: تسجيل الدخول بأدوار مختلفة والتحقق من أن كل مسار متوقع (مبيعات، مشتريات، إعدادات، …) يُرجع 200 وليس 403، وأن المسارات غير المسموح بها تُرجع 403.
   - التحقق من أن الواجهة بعد جلب path-rules تعرض/تخفي عناصر السايد بار بشكل مطابق لـ PATH_PERMISSION_PATTERNS أو أضيق.

### 4.3 توصيات أمنية وعملية

| التوصية | الوصف |
|---------|--------|
| **عدم تفعيل Default Deny في الإنتاج قبل الملء** | إما ملء الجدول أولاً ثم التشغيل بـ Default Deny، أو الإبقاء مؤقتاً على Safe Mode (السماح عند عدم وجود قواعد) حتى اكتمال القواعد. |
| **مزامنة الجدول مع التوثيق** | الحفاظ على تطابق بين `path_permission` وملف توثيق_الصلاحيات.md (جدول ربط المسار بالصلاحية) وتحديث التوثيق عند إضافة مسارات أو صلاحيات. |
| **مراقبة 403** | تفعيل DEBUG لـ PathPermissionFilter ومراقبة طلبات 403 غير متوقعة بعد النشر. |
| **مراجعة Whitelist** | أي مسار عام جديد (مثل APIs للهواتف أو شركاء) يجب إضافته صراحة إلى PathPermissionConstants وإلى SecurityConfig إن لزم. |
| **عدم الاعتماد على الواجهة وحدها** | حماية الـ Backend (path_permission + @PreAuthorize) هي مصدر الحقيقة؛ إخفاء رابط في الواجهة لا يمنع استدعاء API مباشرة. |
| **صلاحيات FRONTEND والـ API** | أنماط FRONTEND يجب أن تعكس نفس سياسة الصلاحيات مثل الـ API للقسم نفسه حتى لا يظهر رابط ثم يُرفض الطلب. |

## 5. الخطوات العملية للمرحلة 5 الكاملة

1. **إضافة Seed (أو سكربت SQL) لـ path_permission:**
   - قواعد PathType=API لجميع المسارات المهمة (حسب القائمة أعلاه) مع ربط كل نمط بـ PermissionID من جدول `permissions`.
   - قاعدة لـ GET `/api/permissions/path-rules` (أو `/api/permissions/**`) بصلاحية يملكها من يحتاج تحميل القواعد.
   - معالجة GET `/api/company` (قاعدة أو استثناء).
2. **إضافة قواعد PathType=FRONTEND** لأنماط مسارات لوحة التحكم وربطها بنفس صلاحيات الأقسام/العناصر في التوثيق.
3. **تشغيل path-audit** والتحقق من خلو `uncoveredPaths` من المسارات المطلوبة.
4. **اختبار الواجهة:** التأكد من أن جلب path-rules ينجح وأن القائمة الجانبية والـ Routes تتصرف كما هو متوقع مع القواعد الجديدة، مع الاحتفاظ بالـ fallback عند الفشل أو القائمة الفارغة.
5. **توثيق التغييرات** في خطة الصلاحيات وتحديث توثيق_الصلاحيات.md (جدول path_permission، صلاحيات الأقسام، مسارات الواجهة).
